👤 What This Feels Like for You (End-Result Overview)

Right now, Adamant records your meetings, transcribes them, and writes AI summaries — but each meeting lives in its own silo. If you want to ask "What did we decide about pricing last month?" you have to manually hunt through meetings or paste transcripts into a chatbot that has no memory of them.

After this plan is built, here's how it works:

When you finish a meeting

Adamant silently creates a wiki article for it. This is not a raw data dump. The LLM reads your full transcript + AI summary + notes, then distills them into a clean, structured page — roughly 500–2000 characters, about as long as a detailed email. It contains only what's worth remembering: key decisions made, action items with owners, people who attended, topics discussed, and specific numbers or dates. The rambling, the "ums," the side chatter — all left behind.

This happens automatically in the background after your meeting saves. You never have to lift a finger, and every meeting just makes your wiki richer.

When you have a question

You open the Memory page (a new Brain icon in the sidebar) and type a question — like "What did we decide about the API redesign?" or "Who owes me action items from last week?"

Instead of dumping raw transcripts into the AI, Adamant does two smart things:

  1. Finds the right meetings. It quickly searches your wiki to pick the 3–5 most relevant articles (not raw transcript fragments — structured, readable pages).
  2. Answers with sources. The AI reads those articles and gives you a clear answer — with clickable citations showing exactly which meetings it drew from.

Over time

  • Every meeting adds a new article. Your wiki grows into a searchable knowledge base of everything discussed.
  • People and topics become interlinked. Ask "What does Alice think about the timeline?" and Adamant knows Alice was mentioned in 4 meetings, reads the relevant ones, and answers.
  • Great answers can be saved back into the wiki, so next time someone asks the same question, the answer is already there.

The raw transcript is not included in the article. The LLM reads it during compilation but only keeps the signal, not the noise. This is what makes the wiki useful — instead of reading 10k chars of raw dialogue, the LLM at query time reads a 500-char article that tells it exactly what it needs to know.

But what about specific details the wiki misses? (e.g., "What was the exact budget Bob proposed?")

When you ask a question, the chat handler runs FTS5 against the raw transcripts as well. It finds small snippets (a few hundred chars) where the user's query terms appear in the original transcript. Those snippets are included alongside the wiki articles in the final LLM answer pass. So the LLM gets both: the big-picture context from the wiki article, and the verbatim detail from the raw transcript where it matters. No extra LLM calls — FTS5 already ran as the routing step.

🎯 Why This Exists

The existing chatbot (chat/handler.rs) already does RAG — it runs FTS5 search against the user's question, fetches raw transcripts/notes/summaries for matched meetings, and dumps them into a 60k-character flat string as LLM context. This works, but has fundamental problems:

  • No structure. Raw transcript dumps are hard for the LLM to reason over — they're fragmented, repetitive, and lack hierarchy.
  • No routing. The LLM reads all context in one shot. There's no "find the right article first, then read it" flow.
  • No citations. The response is a plain string. The LLM guesses which meeting something came from.
  • No compounding. Every question starts from scratch. Answers don't make the system smarter.
  • No cross-linking. People, projects, and decisions mentioned in one meeting are invisible to questions about other meetings.

The fix: After each meeting, an LLM compiles it into a structured markdown wiki article on disk. A global _index.md maps everything. At query time, first try the existing FTS5 full-text search (search/fts.rs:103, <200ms, BM25-ranked, entirely local) to find relevant meeting IDs. If FTS5 finds matches, skip the routing LLM call — go straight to reading those wiki articles and answering. Only fall back to an LLM routing pass when FTS5 returns nothing. This hybrid saves 2-5s of latency on the common path while keeping the LLM routing pass as a safety net for queries FTS5 can't handle.

What is FTS5?

FTS5 (Full-Text Search 5) is SQLite's built-in full-text search engine. Adamant already has it at search/fts.rs:103 — it creates a virtual table meeting_search_fts that indexes meeting titles, transcripts, notes, and summaries (migration 20260327000000_add_search_fts5.sql). When you search, it uses BM25 ranking to score results by relevance, weighting title ×4, transcript ×2, notes ×2, summary ×1. Entirely local, <200ms, no LLM call needed. The plan reuses it as a fast routing shortcut before the expensive LLM routing pass, and as a fallback when the LLM routing fails.

Key architectural decisions

  • Filesystem .md, not SQLite blobs. Articles live in ~/.adamant/wiki/ as plain markdown files — the exact structure Karpathy describes. SQLite tracks only metadata (wiki_metadata table).
  • FTS5 shortcut routing, not pure LLM index navigation. First try FTS5 (<200ms). If it returns matches, skip the routing LLM call entirely — saves 2-5s latency. Only use LLM routing when FTS5 returns nothing. Both paths still read full wiki articles for the final answer.
  • No vector embeddings. Deliberate. At personal meeting-recorder scale (~500 meetings × 50k words), FTS5 BM25 + a well-maintained markdown index outperforms chunked similarity search. Karpathy: "at ~100 articles and 500k words, well-maintained markdown indexes just work."
  • Rust-native, not Python backend. The existing chat/handler.rs proves in-process works. No server dependency.
  • [[concept]] internal links. Wiki articles link to each other using [[Alice Chen]], [[Q3 Launch]] notation. This creates a dense relationship web that compounds over time.
  • Phase-gated. Each of 3 phases ships independently.

Auto-trigger wiki compilation (debounced + stale-checked)

After a meeting's data finishes saving, spawn a background tokio::spawn to compile. Three trigger points:

  1. Recording stop — in api/api.rs:~1246 after refresh_meeting_fts() call on transcript finalization
  2. Notes save — in api/api.rs:~2067 after api_save_note calls refresh_meeting_fts()

Deduplication: All three trigger points can fire within seconds for the same meeting. Maintain a HashMap<meeting_id, CancellationToken>. When a new trigger arrives for a meeting that already has a pending compilation, cancel the old one and schedule a new one with a 5s debounce. Also check is_stale — if the article is already fresh and the content hash hasn't changed, skip compilation entirely.

Emit "wiki-article-ready" Tauri event on success: { meeting_id, title }.

⚠️ Edge Cases & Safety (Reviewed)

  • No transcript yet: A newly created notes-only meeting has no transcript rows. Skip transcript section, compile from notes only. If neither exists, don't write an article.
  • LLM returns empty/malformed article: Retry once with 1s delay. If still fails, set is_stale = 1 and store error in wiki_metadata.error. Log structured context (meeting_id, model, error). Next re-compile-all picks it up.
  • Routing JSON parse failure: Lenient parser strips markdown fences, replaces single quotes, removes trailing commas. If still unparseable, fall back to FTS5-based selection. Validates IDs against wiki_metadata table — silently drops invalid IDs.
  • FTS5 routing returns nothing: Fall back to LLM index routing pass. If that also fails, use old FTS5-based raw transcript selection from chat/handler.rs:116-166.
  • Stale article: A re-transcribed meeting triggers mark_stale. The chat handler optionally checks staleness and re-compiles on demand.
  • Deleted meeting: ON DELETE CASCADE on wiki_metadata removes the metadata row. The .md file stays on disk as a tombstone.
  • Concurrent compilation: 5s debounce per meeting_id (HashMap<meeting_id, CancellationToken>). _index.md uses temp file + atomic rename — no file lock needed, reads never see partial writes.
  • Very large meetings: If meeting exceeds context limits, cap compilation at max context and note truncation in the article.
  • Model change: Wiki compiled with old model stays. New compilations use new model. Re-compile-all (stale-only) rebuilds fresh articles.
  • Prompt injection: User content wrapped in XML tags in compilation prompt with defense instruction. (Low risk for a personal tool where the user records their own meetings.)
  • Citation LLM output missing SOURCES line: Return answer with empty citations. Log warning. UI shows "No sources cited."
  • Re-compile-all with 0 stale: Display "All articles up to date." Zero LLM calls.
  • Disk full during compilation: File write fails → return error. Mark is_stale = 1 with error. Log. Next re-compile-all retries after user frees space.
