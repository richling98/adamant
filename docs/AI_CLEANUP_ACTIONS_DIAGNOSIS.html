Engineering diagnosis

AI Cleanup Actions Are Not Reliably Captured

A code-level diagnosis of why the “Weekly Team Meeting” example produces no visible actions, how the current workflow differs from the intended human workflow, and the implementation plan to make action capture dependable.

Repository examined: Adamant. Primary path: native Tauri/Rust summary pipeline, not the legacy FastAPI path. Report date: July 13, 2026.

Primary root causeThe committed extractor does not recognize the heading “next steps and actions”.
Secondary root causeThe committed extractor skips transcript LLM extraction whenever notes yield any items.
Visibility bugActions are saved asynchronously after summary completion, but the UI refreshes at summary completion.
Expected resultThree unchecked actions linked to the Weekly Team Meeting.

Layman’s Overview

Adamant currently uses two different “secretaries”. The first secretary creates polished meeting notes. The second secretary creates the Actions list. The first secretary is explicitly told not to create an Action Items section, so the Actions page depends entirely on the second secretary.

The second secretary currently has two problems. It looks for a narrowly defined heading instead of understanding the meaning of a heading, and older code tells it to stop reading the transcript if it found anything in the notes. In the example, the heading is written as next steps and actions, which is understandable to a person but not an exact match for the current heading list. The bullets therefore never reach the action detector.

The fix is to give one extraction pass both complete sources: all manual notes and the complete transcript. The model should be told to behave like a meeting secretary: identify concrete next steps, commitments, follow-ups, and assigned tasks; ignore facts, takeaways, announcements, and strategy unless they create a specific task. A small deterministic parser can remain as a fallback, but it must never be the gate that prevents the full-source pass.

What Happens Today

  1. The user clicks Generate AI Cleanup in frontend/src/components/MeetingDetails/SummaryGeneratorButtonGroup.tsx.
  2. useSummaryGeneration.handleGenerateSummary fetches every transcript page, obtains live notes or persisted notes, formats the transcript, and invokes api_process_transcript.
  3. frontend/src-tauri/src/summary/commands.rs stores transcript chunks and starts a background summary task.
  4. summary/processor.rs generates the polished Markdown summary. Its system prompt explicitly says “Do not include Action Items.” This is intentional, but it means the summary response is not the source of truth for actions.
  5. After the summary is saved, summary/service.rs spawns a separate best-effort TodoExtractor task.
  6. summary/todo_extractor.rs first parses specially headed note sections, then may call an LLM over the transcript, filters the result, deletes prior extracted rows, and inserts new rows into todos.
  7. The Actions page reads those rows through api_get_all_todos or api_get_todos_by_date. The homepage only provides the navigation entry point; it does not render the individual actions.

Why The Transcript Does Not Rescue This Example

The transcript is unrelated to the meeting notes and contains no commitments such as “I will send”, “we need to schedule”, or “follow up with”. A secretary should correctly ignore it. The actions are unambiguous only because they exist in the manual notes. This is precisely why manual notes must be a first-class input to the same extraction workflow, not an optional pre-filter.

The current transcript prompt also says to extract only explicit personal commitment language. That is a reasonable false-positive safeguard for spoken transcript text, but it is not sufficient for handwritten action lists, where imperative bullets are the explicit task language. The source type needs to be communicated to the extractor.

Recommended Design

Target behavior: when AI Cleanup is requested, run one action-extraction step over the complete manual notes plus the complete transcript. Treat manual action sections and imperative bullets as high-confidence evidence. Treat transcript commitments, assignments, follow-ups, deadlines, and decisions that create work as evidence. Return structured actions with text, owner, deadline, confidence, and source.

Proposed extraction contract

Return only valid JSON:
{
  "actions": [
    {
      "text": "Send a follow-up email to Jane",
      "owner": null,
      "deadline": null,
      "source": "manual_notes",
      "confidence": 0.99
    }
  ]
}

Rules:
- Read both sources completely before deciding.
- Preserve explicit manual-note actions, including imperative bullets.
- Extract concrete tasks, follow-ups, commitments, assignments, and deadlines.
- Do not extract facts, metrics, announcements, takeaways, opinions, or strategy by themselves.
- Do not invent owners, deadlines, or task details.
- Return [] only when no concrete action exists.

For the supplied notes, the expected normalized actions are:

  • Send follow-up email to Jane
  • Send meeting transcript to Ron
  • Schedule next week’s call

Few-shot rules to encode in the prompt

  • Manual notes and transcript are equally important source material, but their language differs: imperative bullets are strong evidence in notes, while explicit commitments and assignments are strong evidence in speech.
  • Extract an action only when the source explicitly states a task, commitment, assignment, follow-up, or deadline.
  • Use surrounding context to resolve references such as “send it”, but do not add details that are not present.
  • Merge duplicate actions across notes and transcript rather than creating duplicate rows.
  • Facts, metrics, announcements, goals, priorities, opinions, and discussion topics are not actions by themselves.
  • When evidence is ambiguous, omit the item. A missed low-confidence implication is preferable to a fabricated task.

User-Only Action Capture

New requirement: an action should be saved only when it is relevant to the Adamant user. A transcript may contain many valid tasks assigned to other attendees, but those tasks must not clutter the user's personal Actions list. This requires speaker attribution and action ownership to be modeled separately from ordinary text extraction.
V1 product decision: use the audio capture path as the speaker distinction. Treat speech from the Adamant user's microphone as adamant_user and speech from system audio as other_participant. This is intentionally a practical convention for the normal remote-meeting setup, not biometric-grade speaker identification. Room recordings with multiple people on one microphone are an accepted v1 limitation.
Transcript display decision: every transcript segment attributed to the Adamant user must include [you] immediately after its timestamp. Example: [00:42] [you] I will send the revised proposal to Maya. Other-participant segments may retain the existing timestamp-only format, or use an internal non-displayed role label for extraction. The visible marker is specifically intended to make the user's own speech obvious when reviewing a recording transcript.

Why microphone versus system audio is useful, but not enough

In a typical remote meeting, the microphone is the Adamant user's voice and system audio contains the other participants. That makes separate audio paths the most practical first signal. However, it is not an identity guarantee:

  • A second person in the room can speak into the user's microphone.
  • Echo, monitoring, or a meeting application can send the user's voice back through system audio.
  • Some setups capture all meeting audio through one input device, so every participant appears as microphone audio.
  • Screen-captured or browser audio can contain media, announcements, or unrelated speech.
  • A user may join from a phone or another device, making the expected channel assignment different.

For v1, the architecture should use the audio source directly: microphone speech is user speech and system-audio speech is other-participant speech. More advanced diarization or voice matching can remain future hardening, rather than blocking the first implementation.

Recommended attribution model

Each transcript segment should carry attribution metadata before it reaches the action extractor:

{
  "text": "I will send the revised proposal to Maya by Friday.",
  "audio_start_time": 42.1,
  "audio_end_time": 47.8,
  "audio_source": "microphone",
  "speaker_id": "speaker_0",
  "speaker_role": "adamant_user",
  "speaker_confidence": 0.96,
  "attribution_method": "channel_prior",
  "display_prefix": "[you]"
}

The formatted transcript sent to the action extractor and shown in the transcript view should preserve the marker:

[00:18] [other] We reviewed the customer pipeline.
[00:42] [you] I will send the revised proposal to Maya by Friday.
[00:51] [other] That sounds good.

[you] is the required user-facing marker. An [other] marker may be used internally or displayed if the product later wants symmetrical speaker labeling, but it is not required for this request.

Recommended values:

  • audio_source: microphone, system, or mixed/unknown.
  • speaker_id: a meeting-local anonymous label such as speaker_0; do not store raw voice embeddings in the transcript or send them to an LLM.
  • speaker_role: adamant_user, other_participant, unknown, or overlap.
  • speaker_confidence: a bounded confidence score from the attribution layer, not the language model.
  • attribution_method: channel_prior, diarization, enrolled_voice_profile, manual_correction, or unknown.

How v1 identifies the Adamant user

  1. Preserve channel provenance. Keep microphone and system audio separate through VAD and transcription, rather than mixing them before Whisper.
  2. Label microphone segments. Emit audio_source: microphone, speaker_role: adamant_user, and a display prefix of [you].
  3. Label system segments. Emit audio_source: system and speaker_role: other_participant. Multiple remote participants can share this role; v1 does not need to distinguish Jane from Ron.
  4. Use manual notes as an independent user-owned source. Treat an unowned imperative in the user's notes as user-owned by default, while excluding note items explicitly assigned to another named person.
  5. Keep an explicit limitation. If multiple people speak into the same microphone, v1 will attribute that speech to the user. This is acceptable for the expected use case and should be documented rather than hidden.
Scope boundary: v1 does not require diarization, voice enrollment, or user confirmation. Those are future options if real-world feedback shows that shared-room recordings create enough false user actions to justify the added model, storage, and privacy complexity.

User-only extraction policy

The action extractor should receive all source material, but it should be given an explicit ownership policy. It should not simply delete every action spoken by someone other than the user, because manual notes can summarize an action the user agreed to perform without using first-person wording.

Ownership policy:
1. Include a transcript action when the attributed speaker_role is adamant_user.
2. Exclude a transcript action when speaker_role is other_participant and the action belongs to that speaker.
3. Exclude actions assigned explicitly to another named person unless the notes explicitly say the user owns or must coordinate that task.
4. Treat an imperative bullet in the user's manual notes with no named owner as user-owned.
5. Treat a named owner in manual notes as authoritative: "Jane: send..." is Jane's action, not the user's.
6. If speaker attribution or ownership is unknown, omit the action rather than guess.
7. Never invent a user action merely because a goal, priority, decision, or topic implies that work should happen.

Recommended implementation phases

  1. Phase 1: preserve channel provenance. Modify the audio pipeline so microphone and system streams are not mixed before separate VAD/transcription branches, or run a parallel source-preserving transcription path. Add source and timestamps to emitted and persisted transcript segments.
  2. Phase 2: add v1 attribution fields and display formatting. Add audio_source, speaker_role, speaker_confidence, attribution_method, and the derived display_prefix to Rust models, API responses, frontend types, exports, and the migration path. The transcript renderer must format user segments as [timestamp] [you] text. The existing speaker column is not enough and currently appears unused. Use channel_prior as the attribution method.
  3. Phase 3: make ownership a first-class extraction field. Return owner_scope as adamant_user, other_participant, or unknown. Persist only adamant_user actions into the personal Actions table by default.
  4. Phase 4: apply the full-source action prompt. Give the LLM the complete manual notes and the complete attributed transcript, including [you] markers. Include microphone/user segments for user-owned action extraction and system/other segments only as context for resolving references. Never let a system-audio action become the user's action merely because it sounds important.
  5. Phase 5: validate the expected setup. Test normal remote meetings, microphone-only speech, system-only speech, mixed source ordering, manual note ownership, and exact visible formatting such as [00:42] [you]. Document shared-room attribution as a known v1 limitation.
  6. Future Phase 6: optional diarization and enrollment. Only if product feedback shows material problems, evaluate the legacy Pyannote-related code, local voice profiles, or user confirmation. These are not prerequisites for the v1 build.

Risks and recommended safeguards

  • False user attribution: v1 uses the documented microphone-as-user channel prior. Do not describe this as biometric identification; diarization or enrollment is future hardening only.
  • Shared-room recordings: v1 attributes microphone speech to the user by design. Document this limitation and measure real-world impact before adding confirmation or a more complex model.
  • Overlapping speech: v1 follows the source channel label. If overlap detection is already available, it may lower confidence, but it must not block the v1 implementation.
  • Privacy: keep voice embeddings local, encrypt them if persisted, provide deletion, and do not include biometric vectors in LLM prompts or analytics.
  • Missing attribution in old meetings: use the conservative legacy policy and label those actions as inferred from notes, not as speaker-verified.
  • Ownership versus responsibility: “send the transcript to Ron” may be the user's task even though Ron is named as the recipient. Parse grammatical responsibility, not just named entities.

Conclusion

This is not primarily an LLM intelligence problem. The supplied actions are clear enough that a human or a simple parser should capture them. The main failure is upstream: the current pipeline can discard the notes before the model sees them, and its exact heading matcher does discard this specific note section. The older shipped branch additionally suppresses transcript extraction under the wrong condition.

Implementing the unified full-source extractor, broadening note-section handling, removing the skip gate, making parsing tolerant but validated, and synchronizing the UI with actual todo persistence will produce the intended result: AI Cleanup behaves like a secretary who reads all meeting material and reports the concrete next steps, while avoiding facts and generic takeaways.
