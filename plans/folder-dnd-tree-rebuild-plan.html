Adamant Folder Drag Alignment Final Plan

Overall Progress: 100% for this final folder visual-alignment pass

Folder drag/drop is now functionally correct, and meeting-note drag/drop alignment feels right. The remaining problem is folder-only visual alignment: the folder highlight or insertion line still activates slightly below where the user visually expects. This plan fixes that by making the visible dragged folder and the geometry used for projection the same thing.

Folder behavior works Meeting-note alignment works Folder visual alignment still off Likely overlay/geometry mismatch

Current Good State

Root reordering, folder nesting, nested-to-root movement, and meeting-note folder drops work. Meeting-note visual feedback is aligned with the dragged meeting note midpoint.

Remaining Bad State

Folder drags still require the visible folder to be slightly lower than the intended target before the target folder highlights or the insertion line appears.

Main Fix

Remove the mismatch between the folder drag visual and the folder drag geometry. The projection point must be the midpoint of the same visible row the user is moving.

Updated Root Cause Analysis

1. Meeting notes align because the visible item is the measured item

Meeting notes use useDraggable to transform the actual meeting row. The user sees the same row whose initial rectangle is used to calculate the moved midpoint. That is why meeting-note folder highlighting now feels centered and intuitive.

2. Folders still use a separate compact drag overlay

Folder rows use useSortable, and the sidebar also renders a compact DragOverlay through FolderDragOverlay. That overlay is not the same height, width, indentation, or row geometry as the original sidebar folder row. The code projects from the original row midpoint, but the user visually tracks the overlay or a transformed sortable row. Those can diverge.

3. The compact overlay can explain the consistent vertical offset

If the visible drag preview is not the same rectangle used for projection, the target highlight can feel slightly above or below the item being moved. The user's report matches that: folder moves work, but the highlight appears only after moving the visible folder slightly lower than expected.

4. The temporary cursor-selection weirdness suggests stale drag state or competing drag visuals

The one-off issue where the cursor had to click slightly below a folder to select it may have been caused by drag overlay state, transformed sortable rows, or incomplete cleanup after a drag. The fix should simplify folder drag visuals and aggressively clear drag state on end/cancel/start so stale overlays cannot affect subsequent interaction.

5. This should be a visual/geometry fix, not another persistence rewrite

The backend move APIs and the folder projection helper now pass the important functional tests. The plan should not touch persistence. It should make folder drag visuals match the geometry that drives the already-working projection.

Desired Behavior

Gesture Expected Behavior Expected Feedback
Drag a folder onto another folder from above. The target folder highlights when the visible dragged folder is actually over it. The blue rectangle appears when the dragged folder row's visible midpoint is in the target row's center band. No need to drag below the target.
Drag a folder onto another folder from below. The same centered rule applies regardless of drag direction. The highlight tracks the visible folder midpoint, not the cursor grab point or a separate overlay.
Reorder folders up or down. Reordering remains functionally correct. The insertion line appears at the boundary implied by the visible dragged folder midpoint.
Drag meeting notes into folders. The existing good behavior remains unchanged. Meeting-note highlight alignment stays centered and intuitive.

Critical Decision

Make folder drags visually match meeting-note drags. The safest fix is to remove the compact folder DragOverlay and let the actual sortable folder row be the visible moving item, just like meeting notes. If a folder overlay is still needed, it must be an exact full-row clone with identical dimensions and midpoint, not the current compact pill.

The preferred implementation is: remove DragOverlay/FolderDragOverlay for folder drags, keep the actual sortable row visible and transformed, and compute projection from that same row's measured midpoint. This aligns the mental model, the visual model, and the geometry model.

Implementation Plan

Step 1: Remove or Replace the Compact Folder Drag Overlay

  • Remove DragOverlay and FolderDragOverlay from the normal folder drag path.
  • Let SortableFolderTreeRow be the actual visual item that moves during drag, mirroring meeting-note behavior.
  • If removing the overlay causes an unacceptable sortable visual, replace it only with a full-width row clone that has the exact same height, padding, indentation, and midpoint as the measured row.
  • Remove now-unused activeDragFolder state if the overlay is removed.

Step 2: Recalibrate Folder Projection Against the Visible Row

  • Keep the active row rect snapshot from drag start.
  • Use getMovedRectCenter(activeRect, delta) only if the visible moving row is the same rect as activeRect.
  • If dnd-kit sortable transforms the active row differently than raw delta, measure the active row's live transformed rect during drag and use that live midpoint instead.
  • Prefer the simplest accurate source: the live visible row midpoint, then the start-rect-plus-delta midpoint as fallback.

Step 3: Harden Drag-State Cleanup

  • Clear folder projection, meeting folder target, active folder id, active drag rect, and row-rect snapshots on every drag start before setting new drag state.
  • Clear the same state on drag end and drag cancel.
  • Guard against stale folderDropProjection being reused if the active drag type is no longer a folder.
  • Keep debug logging for dragVisualCenter, active rect source, and projection so any future offset can be diagnosed quickly.

Step 4: Preserve the Working Meeting-Note Path

  • Do not change meeting-note drag geometry except where shared cleanup requires it.
  • Keep meeting-note folder row highlighting based on the visible meeting row midpoint.
  • Keep meeting notes blocked from folder root insertion lines.
  • Regression-test meeting note drops into root folders, nested folders, and back to Meeting Notes.

Step 5: Add Targeted Tests and Instrumented Manual Checks

  • Keep the existing projection unit tests.
  • Add a focused test for the midpoint helper showing that the visual row center, not overlay dimensions, is the projection coordinate.
  • Add a debug mode manual procedure: enable adamant-debug-folder-dnd, drag folder over target, confirm logged dragVisualCenter.y enters the target rect when the visible row midpoint appears over the target.
  • Run the standard checks: TypeScript, projection tests, Rust folder repository tests, production build, diff check, and app launch.

Step 6: Manual QA Matrix

  • Folder from above into root folder: highlight appears when the visible dragged folder is centered over the target, not below it.
  • Folder from below into root folder: same centered behavior.
  • Folder into nested folder: same centered behavior.
  • Folder reorder above and below: insertion line appears where the visible folder midpoint implies the drop.
  • Nested folder to root above, between, and bottom: behavior remains correct.
  • Meeting note into root folder and nested folder: existing good alignment remains.
  • Click folders after repeated drags: selection/click hit testing is normal and never requires clicking below a folder.

Acceptance Criteria

This pass is complete when folder dragging feels as visually centered as meeting-note dragging does now.
  • Folder highlight appears when the visible dragged folder is on top of the target folder, not when it is slightly below it.
  • Folder insertion lines align with the visible dragged folder midpoint.
  • Folder movement behavior remains correct for reorder, nesting, nested-to-root, and deep nesting.
  • Meeting-note drag/drop behavior and visual feedback remain correct.
  • After repeated drags, normal clicking/selecting folder rows remains aligned with the cursor.
